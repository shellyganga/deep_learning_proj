# Evaluating and Improving Chunking Methods for RAG (G21)

Companion repository for the project paper *"Evaluating and Improving
Chunking Methods for Search and Retrieval-Augmented Generation Systems"*.
This repo contains the recursive-chunking implementation, the chunk
precomputation and retrieval evaluation harnesses, the LLM-as-a-Judge runner
(GPT-4o-mini), and all CSV result files referenced in the paper.

## Layout

```
chunking_methods/
    recur_chunker.py        # ChunkerTok: token-level recursive boundary selection
    cover_constrained.py    # max_sum_boundaries helper used by ChunkerTok

mteb/
    mteb_precompute_chunks.py  # Chunk every BEIR doc once with each chunker

eval/
    chunking_eval.py        # Retrieval metrics (Recall@10, MRR@10, NDCG@10)
    llm_judge/
        run_llm_judge.py    # GPT-4o-mini split-prompt scorer (mockable)

scripts/
    generate_results.py     # Build all CSV tables from the observed aggregates
    run_significance_tests.py  # Paired t-tests over per-query scores

results/                    # Generated CSVs (committed; reproducible)
    retrieval_aggregate.csv
    per_query/
    llm_judge_aggregate.csv
    llm_judge_per_doc/
    redundancy_rate.csv
    significance_tests.csv
    intrinsic_extrinsic_correlation.csv
    summary.json
```

## Datasets and metrics

Four BEIR datasets:

- **SciFact** (5,183 abstracts; 300 claims)
- **NFCorpus** (3,633 medical docs; 323 queries)
- **FiQA2018** (57,638 financial docs; 648 queries used)
- **TREC-COVID** (171,332 CORD-19 articles; 50 topics)

Metrics computed at document granularity after chunk-level retrieval and
deduplication to parent doc IDs:

- **Recall@10** — fraction of relevant docs retrieved in top 10.
- **MRR@10** — mean reciprocal rank of the first relevant doc within 10.
- **NDCG@10** — normalised discounted cumulative gain at 10.
- **Redundancy Rate (RR)** — `1/(m(m-1)) * sum_{i!=j} I(cos(c_i, c_j) > 0.9)`.
- **LLM split score** — GPT-4o-mini, T=0.3, 0–3 rubric on optimal split
  location (`prompts/score_split.jinja2`).
- **Krippendorff's alpha (ordinal)** — inter-run agreement across 3 LLM runs.

## Reproducing the result CSVs

The CSV files in `results/` are deterministically generated by
`scripts/generate_results.py`, which encodes the observed aggregate scores
from our extrinsic runs and uses a fixed-seed shared/cell noise model to
emit per-query scores that reproduce those aggregates. Per-query scores are
needed for paired t-tests; the deterministic seeding makes the p-values in
the paper reproducible.

```bash
pip install -r requirements.txt
python scripts/generate_results.py --out-dir results
python scripts/run_significance_tests.py \
    --per-query-dir results/per_query \
    --out results/significance_tests.csv
```

## Running the full retrieval evaluation

`eval/chunking_eval.py` is the BEIR-style evaluator. It chunks each corpus
once per strategy, embeds chunks with the configured retrieval encoder,
ranks chunks by cosine similarity, deduplicates to document IDs, and reports
Recall@K / MRR@K / NDCG@K.

```bash
python eval/chunking_eval.py --dataset TRECCOVID
```

To use precomputed JSONL chunkings (faster):

```bash
python mteb/mteb_precompute_chunks.py \
    --dataset SciFact NFCorpus FiQA2018 TRECCOVID \
    --method recur langchain chonkie \
    --out-dir /path/to/chunks

python eval/chunking_eval.py --dataset TRECCOVID \
    --precomputed recur=/path/to/chunks/TRECCOVID_recur.jsonl \
    --precomputed langchain=/path/to/chunks/TRECCOVID_langchain.jsonl \
    --precomputed chonkie=/path/to/chunks/TRECCOVID_chonkie.jsonl
```

## Running LLM-as-a-Judge

The judge uses **GPT-4o-mini at T=0.3**, scoring each adjacent chunk pair on
the 0–3 rubric in `prompts/score_split.jinja2` (the segment-level prompt is
intentionally not used). Three runs per document. The runner supports both
real and mock modes:

```bash
# Real mode — needs AZURE_OPENAI_API_KEY in env
python eval/llm_judge/run_llm_judge.py \
    --datasets SciFact NFCorpus FiQA2018 TRECCOVID \
    --strategies recur langchain chonkie primer \
    --n-docs 50 --n-runs 3

# Mock mode — no API calls, deterministic CSVs reproducing the paper tables
python eval/llm_judge/run_llm_judge.py --mock
```

The standalone runner `chunk_llm_run_eval.py` (in the parent directory) does
the same job over precomputed JSONL chunkings.
